• Sharing Study Data Overview for MPS Administrators

    • About sharing study data:
      • Currently, all sharing options are setup by MPS administrators (there are discussions occurring to change this)
      • Access is setup by assigning permissions to users and by assigning groups to studies in the admin interface (this may change - a prototype to allow changing group access on the front end is on stage)
      • Access is granted to a study and everything contained within (no option for finer-grained sharing of parts of a study)
      • There are three sections of the study page (in the admin interface) that deal with sharing: the Study Data Group and Access Group Info section, the Change Tracking section, and Assay Study Stakeholders section

    • Examining sharing subsections:
      • The Change Tracking section:
        • Signed Off By - If this has a user name in it, it means that the study is following the sign-off, acknowledgement, embargo process
        • A sign-off does not open the study to the public, even if there is no stakeholder
        • If there is no stakeholder, the study is open to Access Groups
      • Study Data Group and Access Group Subsection
        • Data Group - This is the team that can add/edit/remove a Study (until the Data Group Administrator signs off the Study)
        • When adding/editing a Study in the front-end, a user who has been added to one or more Data Groups (by an MPS Admin) can select the Data Group for the Study (selection options will be limited to the Editing Groups to which the user is a member).
        • Restricted - Until this is unchecked, a study is closed to the public (it can be unchecked by an admin at any time)
        • Studies in the embargo period (after sign-off and acknowledgement) will be unchecked one year and two weeks after sign-off (automated process), UNLESS the study is 'locked'
        • Keep Private Indefinitely (Locked) - Locking a study stops the study from being released to the next level (blocks automatic migration to public access - public if acknowledged, access if stakeholders, stakeholders if signed off).
        • Access Groups - Any Groups in the 'Choosen Access Groups' box can see the study after stakeholder acknowledgement, or after sign-off if there are no stakeholders
        • Collaborator Groups - Any Groups in the 'Chosen Collaborator Groups' box can see this study, at any time
        • Collaborator Groups are a SPECIAL feature that allows sharing a study regardless of where the study is in the sign-off and embargo process (it is the ONLY feature, currently in the database, that overrides other permissions)!!!!
      • Assay Study Stakeholders Subsection
        • Any number of Stakeholder Groups can be added
        • When a Study is signed off, members of any present Stakeholder Groups will receive an email telling them to look at and approve the study

    • Help Notes:
      • What happens if the 'Use in Calculations' box is check? The study becomes available in the Compound Report.
      • Collaborator groups are completely controlled by the admins (no front-end GUI).
      • To open a study to the public immediately (i.e. bypass sign-off, acknowledgment, embargo) must by done by the admin (uncheck the restricted box), but data providers can check the 'Use in Calculations' box.
      • While the use is sometimes indicated in the group name, there is no real difference between groups (data, stakeholder, access, collaborator). Use caution. Even though a data group and a stakeholder group is associated with the same institution, the users in the groups maybe different.

  • Adding New Data Groups

    • A Group (Data, Stakeholder, Access) is actually a trio of groups. The following is an example for Data Group Rusyn_TAMU
      • Rusyn_TAMU - this is the editor group, sometimes referred to as the group proper
      • Rusyn_TAMU Admin - admins can edit and sign-off (or acknowledge for TCD groups) studies
      • Rusyn_TAMU Viewer - can view studies prior to sign-off (or acknowledgement for TCD groups)

    • A Collaborator Group is two groups, one, the group proper, and one for viewers. The following is an example for the SQL-SAL_1.5 group
      • SQL-SAL_1.5 - add this Collaborator Group to studies to share
      • SQL-SAL_1.5 Viewer - add the collaborators to this group

    • A Stakeholder Group is three groups similar to Data Groups. Thus far, when naming these we have added "TCD_" as a prefix and the name of the institution as a suffix. The following is an example for the SQL-SAL_1.5 model
      • TCD_SQL-SAL_Pitt - Add this group to studies for study review and sign-off
      • TCD_SQL-SAL_Pitt Admin - add users to this group to allow those users to view any studies with this group as a Stakeholder once the study is signed off
      • TCD_SQL-SAL_Pitt Viewer - add users to this group to approve signed off studies

    • To create and prepare a new Group
      • Go to the MPS-Db admin (https://mps.csb.pitt.edu/admin/)
      • Go to the MPS Authentication and Authorization Section
      • Click on Groups and scroll down to review how groups are named
      • Click on the ADD GROUP button on the top right side
      • Give the group a name (for example: my_group)
      • Click the Save and add another button
      • Give the group a name (for example: my_group Admin)
      • Click the Save and add another button
      • Give the group a name (for example: my_group Viewer)
      • Click the SAVE button
      • Return to the admin home page (https://mps.csb.pitt.edu/admin/)
      • Go to the MPS CENTERS AND MODELS Section
      • Scroll to find the Center with which the new group will be associated
      • Click on the center Name to select the center
      • Scroll down to the Groups section and add the new group (the group proper) to the center
      • Scroll down and click the SAVE button

    • Help Notes:
      • It is very easy to forget to attach the group proper to a center. If you forget, you will not be able to use the group.
      • Although Groups are classified into three different types, each Group is fundamentally the same in the backend. The differences are purely functional based on where a Group is used.

  • Assigning Groups to Studies

    • To assign groups to a Study
      • Go to the MPS-Db admin
      • Go to the MPS ASSAYS Section
      • Click on Studies
      • Scroll through the study list and locate the study of interest
      • Click to open the study
      • Scroll to the group section of interest (Data, Access, Collaborator, Stakeholder)
      • Select group(s) for each section (note: the Data Group is selected when the study is added, but it could be edited here)
      • Add the group proper (not the admin or viewer groups) to the study
      • Save the changes

    • Help Notes:
      • Best practice would be to assign appropriate groups (Stakeholder and Access) at the time of study creation.
      • Data Group Admins and Editors can create a study, but cannot add/edit special permission groups to a study
      • Collaborator groups bypass the tri-level sign-off, review, and release protocol
      • Data Providers must work with MPS-Db Admins to set up appropriate groups when/if needed.
      • Clicking on the button to Change Group Access for this Study from the Study Matrices and Items List page will route the user to the Change Study page of the GUI Admin.
      • IMPORTANT: studies should always be assigned to the group proper, that is, the editor group (not admin or viewer)!

    • To quickly see what studies are in a Collaborator Group
      • Go to the MPS-Db admin
      • Go to the Authentication and Authorization Section
      • Click on Groups
      • Look in the Collaborator Group Display column
      • Scroll through the groups or use the search bar
      • Click on the Show/Hide Collaborator Studies link

  • Assigning Users To Groups (Data, Stakeholder, Access, and Collaborator)

    • To assign users to a Group
      • Go to the MPS-Db admin
      • Go to the Authentication and Authorization Section
      • Click on Users
      • Scroll through the users or use the search bar
      • Click to open the user details
      • Change group membership as needed
      • Save the changes

    • Help Notes:
      • Users should be assigned to appropriate groups when they register
      • Users may have different roles in different groups
      • All users of collaborator groups should be Viewers

  • Study Component Add

    In the past, users had to be assigned the "Add X Front" and/or "Change X Front" (where X is Cells, Compounds, etc.) to add study components in the front-end. Now, any user who is a memeber of at least one Group (with the role of Editor or Administrator) can add study components in the front-end.

  • Study Component Edit

    Allowing study components to be edited is tricky and was the subject of much debate.

    • Editing Study Components:
      • MPS Admins can edit study components via the admin interface
      • Other Users - to edit an MPS Model, a user must be an editor or administrator of, at least one Group associated with the Model's Center.
      • Other Users - to edit a specific Study Component, other than an MPS Model, the user must have either:
        • created that specific study component, or
        • the specific study component must not be in use (for example, if the specific study component is a Device, to be editable, the Device must not have been used in an MPS Model in the database)

    • Editing Help Notes:
      • Some components, like Methods and Targets, have a 'limited' interface to allow for binding Target and Methods to each other, regardless of who created the specific study component or whether the specific study component is in use
